üß† What is a skill?

A skill is a saved set of instructions that Claude Code follows every time you type a specific slash command. Think of it as writing a really detailed sticky note for your AI colleague ‚Äî except this one actually reads it.

Instead of typing "Please review this file for security issues, check for SQL injection, XSS, hardcoded secrets, and give me a score out of 10 with specific line numbers‚Ä¶" every single day, you write that once in a file and forever after you just type: 

/security-review src/auth.py

That's it. Your coffee is getting cold. Go get it.

ü™Ñ The actual magic: Skills aren't just shortcuts ‚Äî they are context for Claude. A skill can tell Claude what persona to take, what to check for, what NOT to do, what tools it can use, and how to format its output. It's the difference between asking a random intern for help and briefing a specialist.

Skills vs. Subagents ‚Äî don't mix these up

Claude Code has two related but distinct concepts:

  • Skills ‚Äî files you create. They run inside your current conversation. Triggered with /skill-name.
  • Subagents ‚Äî independent Claude instances that Claude spins up internally using the Task tool. You don't create these yourself; Claude fires them up when a task is complex enough. Add context: fork to a skill's YAML and it will run inside a subagent (isolated context, no cross-contamination).

This guide covers skills. (Subagents are covered in 02_subagents.html.)

üóÇÔ∏è Where do skills live?

Skills live in folders named after them. Each folder contains at minimum a SKILL.md file. There are two places you can put them, with different scopes:

Option A ‚Äî Project skill (shared with your team)

Put the skill inside your project's .claude/skills/ folder, then commit it to git. Everyone on the team gets the same commands automatically. No onboarding doc needed ‚Äî the skill is the documentation. 

# Create the folder for your skill
mkdir -p .claude/skills/my-skill
# PowerShell ‚Äî create the folder for your skill
New-Item -ItemType Directory -Force -Path ".claude\skills\my-skill"

Then commit it:

git add .claude/skills/
git commit -m "feat: add my-skill slash command"

Option B ‚Äî Personal skill (follows you everywhere)

Put the skill in your home directory. It will be available in every project on your machine. Great for personal habits like /summarize, /explain-like-im-five, or /rubber-duck. 

mkdir -p ~/.claude/skills/my-skill
New-Item -ItemType Directory -Force -Path "$env:USERPROFILE\.claude\skills\my-skill"

üî¨ Anatomy of a SKILL.md

Every SKILL.md has two parts separated by ---:

Part 1 ‚Äî YAML frontmatter (the config)

This sits at the very top between two sets of --- dashes. It tells Claude Code the metadata about your skill: what it's called, when to use it, and any special behaviour flags. 

---
name: my-skill             # becomes /my-skill
description: Use this when‚Ä¶  # tells Claude when to auto-invoke it
---

Part 2 ‚Äî The instructions (plain English)

Everything after the second --- is the actual instructions Claude will follow. Write it like you're briefing a very capable but literal-minded contractor. No vague vibes ‚Äî specific steps, clear output format, explicit edge cases. 

# What to do

When this skill is invoked, do the following:

1. First, read the file(s) mentioned in $ARGUMENTS
2. Then, perform your analysis
3. Finally, present results in the format described below

# Output format

- Give a score out of 10
- List issues with file name and line number
- Be direct. No fluff.

The $ARGUMENTS variable

When someone types /my-skill foo bar, the text after the skill name (foo bar) is available in your instructions as $ARGUMENTS. 

/review src/auth.py strict
         ‚Üì inside your SKILL.md ‚Üì
$ARGUMENTS = "src/auth.py strict"

üìã The Full Template

Copy this, rename it, fill in the blanks. You're 90% done before you even start. 

---
name: your-skill-name
description: >
  Use this skill when the user wants to [primary use case].
  Do NOT use this skill when [exclusion case].
---

# Your Skill Name

You are a specialist in [domain]. Your job is to [main purpose].

## When invoked

1. Read the relevant files (use `$ARGUMENTS` as the target, or ask if not provided)
2. Step 2 ‚Äî the main action
3. Step 3 ‚Äî follow-up action
4. Present your findings in the format below

## Output format

Describe exactly how you want Claude to format its response.
The more specific, the more consistent the output will be.

## What to check / focus on

- Point 1
- Point 2
- Point 3

## Do NOT

- Anti-pattern 1 you want to avoid
- Anti-pattern 2
üé≠ Confession: The description field is the most important and most neglected part of a skill. It determines when Claude auto-triggers it. "Use when reviewing code" is bad. "Use when the user explicitly asks for a code review of a specific file or function. Do NOT use for general questions about code." is good. Put in the effort ‚Äî it pays off every single time.

üöÄ A Real Working Example

Here's a complete skill you can copy, save, and immediately use. It does code reviews with a score, specific issues, and line numbers.

Step 1 ‚Äî Create the folder

mkdir -p ~/.claude/skills/review
New-Item -ItemType Directory -Force -Path "$env:USERPROFILE\.claude\skills\review"

Step 2 ‚Äî Create the SKILL.md

Save this as ~/.claude/skills/review/SKILL.md%USERPROFILE%\.claude\skills\review\SKILL.md: 

---
name: review
description: >
  Use when the user asks for a code review of a specific file, function, or snippet.
  Reviews for correctness, security, style, and performance.
  Do NOT use for general questions about how code works.
---

# Code Review

Review the code in $ARGUMENTS (or the current file if no argument given).

## Check for

- **Correctness** ‚Äî does it do what it claims?
- **Security** ‚Äî SQL injection, XSS, hardcoded secrets, unsafe deserialization
- **Style** ‚Äî is it readable? Would a new team member understand it in 30 seconds?
- **Performance** ‚Äî obvious bottlenecks, N+1 queries, unnecessary re-renders

## Output format

1. **Score** ‚Äî X/10 with one sentence of justification
2. **Critical issues** ‚Äî list with file:line, severity (üî¥ high / üü° medium / üü¢ low), and fix
3. **Minor suggestions** ‚Äî optional improvements, no pressure

Be direct. Be specific. No padding.

Step 3 ‚Äî Use it

# Review a specific file
/review src/auth.py

# Review with a focus hint
/review src/auth.py focus on the token validation

# Review whatever file is open / being discussed
/review
‚úÖ That's literally it. No boilerplate. No configuration files. No npm install. Just a folder and a markdown file. Skills are refreshingly low-tech for something this powerful.

‚öôÔ∏è All YAML Options

The frontmatter supports more than just name and description. Here's the full list:

FieldRequired?What it doesExample value
name Required The slash command. name: review ‚Üí /review review
description Required When Claude should auto-invoke this skill. Be explicit. "Use when user asks for code review‚Ä¶"
disable-model-invocation Optional Set to true to prevent Claude from invoking it automatically. Only you can trigger it manually. true
context Optional Set to fork to run the skill inside an isolated subagent (no main context pollution). fork
allowed-tools Optional Restrict which tools this skill can use. Comma-separated. Read, Grep

The safety combo ‚Äî for dangerous skills

Any skill that touches production, databases, or external services should use both of these:

---
name: deploy
description: Deploy to production. Only invoke manually, never auto-trigger.
disable-model-invocation: true    # Claude cannot auto-fire this
---

Before doing anything, ask: "Are you sure you want to deploy to production? (yes/no)"

Only proceed if the user explicitly confirms with "yes".

Then:
1. Run tests: npm test
2. Build: npm run build
3. Deploy: npm run deploy:prod
4. Confirm deployment status and report back

The read-only analyst combo

For skills that should only read files ‚Äî never modify anything:

---
name: audit
description: Audit the codebase for security issues. Read-only, no changes.
allowed-tools: Read, Grep, Glob   # strictly read-only
---

Audit the codebase for security vulnerabilities.
You may ONLY read files. Do NOT modify anything.

üí° Tips & Tricks

1. The personal skills folder is underrated

Skills in your home directory follow you to every project. Some personal classics worth creating:

CommandWhat it does
/summarizeSummarize what we've done in this session so far
/eli5Explain the last thing like I'm 5 (no jargon)
/rubber-duckJust listen while I explain a problem. Don't solve it. Ask clarifying questions.
/standupWrite my daily standup based on recent git commits
/refactorRefactor this code. Explain every change you make.

2. Write the description like a legal contract

"Use when reviewing code" will cause Claude to randomly invoke your review skill during conversations about code. Be specific: what triggers it, what definitely does not trigger it.

‚ö†Ô∏è Bad: description: Use for code review
‚úÖ Good: description: Use ONLY when user explicitly asks to review a specific file or function. Do NOT use for general questions about code quality or architecture.

3. Reference files in your skill folder

You can put reference documents in a references/ subdirectory and tell Claude to read them. Great for style guides, API documentation, or custom coding standards your team follows. 

# In your SKILL.md instructions:
Before starting, read the file at references/our-style-guide.md
and follow those conventions strictly.

4. Iterate like a recipe

Your first skill doesn't need to be perfect. Start with 5 lines, use it a few times, then add instructions for the edge cases you encounter. You're teaching it like a new hire ‚Äî gradually, based on actual mistakes, not hypothetical ones.

5. Name your skill folder and the name field identically

The folder name and the name: field in the YAML should match. It keeps things predictable and prevents the kind of subtle bugs that make you question your life choices at 11pm. 

# Good ‚Äî folder name matches name field
.claude/skills/review/SKILL.md  ‚Üí  name: review   ‚úÖ

# Confusing ‚Äî mismatched names
.claude/skills/code-check/SKILL.md  ‚Üí  name: review   ‚ùå

6. Check your skills into git

git add .claude/skills/
git commit -m "feat: add team skills for review, deploy, and audit"

Your skills are documentation, tooling, and onboarding guide all in one. Don't keep them only on your machine.
