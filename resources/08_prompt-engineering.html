üó∫Ô∏è Give Context, Not Just Commands

The single biggest upgrade to prompt quality is adding context. AI models don't know your stack, your constraints, or your codebase's weird history. Tell them.

Bad vs. Good: adding context

‚ùå Too vague

Write a function to get users from the database.

‚úÖ With context

Write a TypeScript function that fetches users from a Prisma PostgreSQL database. It should accept optional filters for `isActive` and `createdAfter`. Return an array of `User` objects (the Prisma type). Use async/await. No raw SQL.

‚ùå No constraints

Refactor this component.

‚úÖ With constraints

Refactor `UserCard.tsx` to split the avatar logic into a separate `Avatar` component. Keep the same props interface. Don't change the CSS classes. Add JSDoc comments to the new component.

The context checklist

For any non-trivial coding request, consider including:

  • Language & framework. "TypeScript in a Next.js 14 app router project."
  • Libraries in use. "We use Prisma, not raw SQL. Zod for validation."
  • File paths. "In src/lib/auth.ts."
  • What exists already. "There's already a getUser(id) function at line 42."
  • What NOT to do. "Don't add new dependencies. Don't change the public API."
  • Why you're doing this. "We're migrating from JWT to session-based auth because..."

üìê Control the Output Format

Unformatted AI output is like getting IKEA furniture instructions as a wall of unbroken prose. A few words of direction yield dramatically different (better) results.

Request specific formats

# Ask for structured output
Review this code and give me:
1. A score out of 10 with one sentence of justification
2. Critical issues (file:line, severity, fix)
3. Minor suggestions (optional, mark clearly)
No padding. No intro paragraph. Start with the score.
# Ask for just the code, no explanation
Write the TypeScript function. Return only the code. No explanation needed.
# Ask for explanation first, then code
First explain the approach you'll take in 2-3 sentences.
Then write the implementation.
Then list any assumptions you made.

Format modifiers that actually work

Add to your promptEffect
Be concise.Removes unnecessary padding and repetition
No explanation needed.Returns just the code, nothing else
Explain your reasoning.Shows the thought process before the answer
Use a table.Structures comparisons into a readable grid
Use bullet points.Breaks down lists instead of prose paragraphs
Return only the changed lines.Avoids returning the entire file for small edits
Include line numbers in your output.Makes it easy to find where changes apply

üé≠ Use Personas and Roles

Telling an AI to take on a role is not a magic trick ‚Äî it's a shortcut for providing a lot of context all at once. "You are a senior security engineer" isn't roleplay, it's a configuration command.

Useful role prompts for developers

# For security-focused review
You are a senior application security engineer. Review this code for vulnerabilities.
Be thorough. Cite OWASP categories where applicable. Assume the worst about user input.
# For performance optimization
You are a database performance specialist. Analyze these queries.
Focus on: missing indexes, N+1 patterns, and full table scans.
Suggest specific fixes with estimated improvement.
# For code review as a skeptic
You are a skeptical senior engineer doing code review.
Your job is to find the edge cases and failure modes that the author missed.
Don't be encouraging. Just find the problems.
# For architecture decisions
You are a staff engineer who has seen many systems fail at scale.
Evaluate this architecture. What are the failure modes?
What will cause pain in 2 years? What would you do differently?
üé≠ "You are an expert" ‚Äî technically every AI prompt is already this. But naming the specific domain ("security engineer", "database performance specialist") narrows the focus in a way that consistently improves output quality. It's cargo-cult prompting that happens to work.

‚õìÔ∏è Chain Your Prompts

One long prompt that tries to do everything is like one meeting that tries to do everything. Everyone leaves confused and nothing gets done.

Break complex tasks into a sequence of focused prompts. Each one builds on the last.

Example: building a feature step by step

# Step 1: Plan
"I want to add a rate limiter to our API. We use Express and Redis.
What's the architecture? Describe the approach ‚Äî don't write code yet."

# Step 2: Implement the core
"Good. Now implement the rate limiter middleware based on that plan.
Use the `ioredis` client we already have at src/lib/redis.ts."

# Step 3: Add tests
"Now write unit tests for the middleware using Vitest.
Mock the Redis calls. Cover: allowed request, rate-limited request, Redis failure fallback."

# Step 4: Integrate
"Apply the middleware to all routes in src/app/api/. Use a limit of 100 requests/minute."

# Step 5: Review
"Review everything you've written. Are there any edge cases or security issues?"

The three-phase pattern

PhasePromptWhy
Plan "Describe the approach. Don't write code yet." Catch misunderstandings before code is written
Implement "Good. Now implement it based on that plan." Code with aligned expectations
Review "Review what you wrote. Any problems?" Catch issues while the context is hot
‚úÖ "Don't write code yet" is one of the most powerful phrases in prompt engineering. It forces the AI to think before acting, and lets you correct the direction before you have a full implementation to undo.

üêõ Prompts for Debugging

The secret to effective AI debugging isn't asking "why is it broken?" ‚Äî it's giving enough information that the AI can actually figure it out.

The complete bug report template

I have a bug. Here's everything:

**What I expected:** [describe expected behaviour]
**What actually happens:** [describe the bug, including exact error messages]
**Relevant code:** [paste the function/file]
**Steps to reproduce:** [what triggers it]
**What I've already tried:** [list your failed attempts]
**Environment:** [Node version, OS, library versions if relevant]

Please:
1. Identify the root cause
2. Suggest a fix
3. Explain why it was happening

For "it works on my machine" bugs

This works locally but fails in production/CI. Help me identify the environmental difference.

**Local:** [Node 22, macOS, .env has X=foo]
**Production:** [Node 20, Linux, environment variable X is not set]
**Error:** [paste the stack trace]

What could cause this discrepancy?

For silent failures

This code runs without errors but produces the wrong output.

Input: [paste example input]
Expected output: [what it should return]
Actual output: [what it actually returns]
Code: [paste the function]

Walk through the execution step by step and find where it diverges.
‚ÑπÔ∏è Paste the actual error message. Not "I'm getting an error." Not "it throws something." The exact message, the exact stack trace. AI models are extremely good at reading stack traces ‚Äî but only if they can see them.

üö´ Anti-patterns to Avoid

The vague mega-request

‚ùå Anti-pattern

Build me a full authentication system with login, registration, password reset, email verification, OAuth with Google and GitHub, rate limiting, and session management.

‚úÖ Better

Let's build auth step by step. Start with email/password login only. We'll add OAuth and other features after the core is solid. First: what's your recommended approach for our Next.js + Prisma stack?

Accepting the first response

‚ùå Anti-pattern

[Copy-pastes AI code without reading it]

‚úÖ Better

"Good. Now review what you just wrote. Is there a null pointer risk in the loop? Does this handle an empty array?"

Not reading the code

AI code is not tested until you test it. It will look correct. It will often be correct. It will occasionally be subtly wrong in a way that only shows up in production at 2am. Read it. Run it. Review it.

Over-specifying the solution

‚ùå Anti-pattern

"Write a for loop that iterates from 0 to array.length, checks if array[i] is greater than 10, and if so pushes it to a new array named result."

‚úÖ Better

"Filter this array to only include values greater than 10."

State the goal, not the implementation. You'll get cleaner code and might learn a better approach.

Asking for too much at once

Each prompt should have one clear goal. "Write the function, add tests, update the docs, and refactor the related code" is four tasks. The AI will attempt all four, do each one shallowly, and you'll spend more time reviewing than if you'd done it yourself.

üìã Reusable Prompt Starters

Copy these, customize, use immediately.

Code generation

Write a [TypeScript/Python/etc] function that [does X].
Context: [stack, libraries, file location].
Constraints: [no new deps / match existing style / don't change public API].
Return only the code.

Code review

Review this code. Check for: correctness, security (injection/XSS/secrets),
performance (N+1/unnecessary work), and readability.
Score it 1-10. List issues with file:line and severity.

Explain unfamiliar code

Explain this code to me.
Assume I know [language] but not [specific library/pattern].
Use plain English. Point out anything unusual or non-obvious.

Add tests

Write tests for this function using [Vitest/Jest/pytest].
Cover: happy path, edge cases (empty input, null, large values), and error cases.
Use the same patterns as existing tests in the codebase.

Refactor

Refactor this code to [be more readable / remove duplication / separate concerns].
Constraints: same public API, no new dependencies, TypeScript only.
Explain each change you make and why.

Architecture decision

I need to decide between [option A] and [option B] for [problem].
Context: [stack, scale, team size, constraints].
Give me a pros/cons table, your recommendation, and one risk I might be underestimating.

Rubber duck debugging

I'm going to explain a problem. Don't solve it yet ‚Äî just ask me clarifying questions
until we've diagnosed the root cause together.
