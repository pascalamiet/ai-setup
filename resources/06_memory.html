üß† What is CLAUDE.md?

CLAUDE.md is a Markdown file that Claude Code automatically reads at the start of every session. It's your project's briefing document ‚Äî the context that Claude would otherwise have to rediscover by reading through your entire codebase every time.

Think of it as the onboarding doc you wish you'd written for your last new hire, except this one actually gets read.

  • üìã Project conventions. "We use snake_case. We never abbreviate variable names. Tabs, not spaces." (Or spaces, not tabs. We won't judge.)
  • ‚å®Ô∏è Commands to know. "Run tests with npm test. Build with npm run build. Deploy requires 2FA."
  • üó∫Ô∏è Architecture overview. What the folders mean. What the main modules do. Where the scary code lives.
  • üö´ Things Claude must never do. "Never edit the generated/ folder. Never commit directly to main."
üé≠ Without CLAUDE.md: "Claude, as I've mentioned before, we use Prisma, not raw SQL, and our API routes are all in src/api/v2, not v1, which is deprecated but still exists for legacy reasons and‚Äî"

With CLAUDE.md: Claude already knows all of that before you type a single word.

üóÇÔ∏è Where to Put It

CLAUDE.md files are read in layers ‚Äî from broadest to most specific. More specific files add to (rather than replace) the broader ones, so you can compose context across levels.

The scope hierarchy

Global
~/.claude/CLAUDE.md
%USERPROFILE%\.claude\CLAUDE.md
Personal preferences. Available in every project, everywhere.
Project
CLAUDE.md
Project root. Commit to git. Your whole team gets it.
Subfolder
src/api/CLAUDE.md
Module-specific context. Loaded when Claude works in that folder.

Create the files

# Project-level (most important ‚Äî commit this)
touch CLAUDE.md

# Personal/global (your preferences across all projects)
mkdir -p ~/.claude && touch ~/.claude/CLAUDE.md

# Subfolder (for module-specific context)
touch src/api/CLAUDE.md
# Project-level (most important ‚Äî commit this)
New-Item -ItemType File CLAUDE.md

# Personal/global (your preferences across all projects)
New-Item -ItemType Directory -Force "$env:USERPROFILE\.claude"
New-Item -ItemType File "$env:USERPROFILE\.claude\CLAUDE.md"

# Subfolder (for module-specific context)
New-Item -ItemType File src\api\CLAUDE.md
‚úÖ Start with one file. Put a CLAUDE.md in the project root and commit it. That's it. You don't need global files or subfolder files until you feel the need for them. Premature context-splitting is a real thing.

‚úçÔ∏è What to Write In It

Here are the categories that consistently pay off. Not all projects need all of them ‚Äî start with what's bitten you before.

Project overview

One paragraph. What the project does, what stack it's on, and what makes it unusual. If someone brand new to your codebase asked "what is this?", answer that.

Essential commands

The commands Claude will want to run: test, build, lint, migrate DB, start dev server. No hunting around for them.

Architecture overview

How folders map to concepts. What src/ contains vs lib/. Whether you have a monorepo. What shared/ means. A 5-bullet list goes further than you'd think.

Coding conventions

Naming conventions, file structure rules, whether you use types everywhere, error handling patterns. Anything that code review would otherwise catch.

Do not touch list

Things Claude should never, ever do. Generated files. Legacy modules. Specific branches. Anything that has caused pain before.

Key environment notes

What env vars are needed. Where .env.example is. How to get credentials for dev.

üìã Full Template

Copy this and fill in the blanks. Delete the sections you don't need.

# Project: [Your Project Name]

## Overview
[One paragraph: what this does, what stack it's on, what makes it notable.]

## Essential Commands
- Install dependencies: `npm install`
- Run tests:           `npm test`
- Run tests (watch):   `npm run test:watch`
- Build:               `npm run build`
- Start dev server:    `npm run dev`
- Lint:                `npm run lint`
- Database migrate:    `npx prisma migrate dev`

## Architecture
- `src/app/`      ‚Äî Next.js app router pages and layouts
- `src/lib/`      ‚Äî Shared utilities and helper functions
- `src/api/`      ‚Äî API route handlers (REST, all v2)
- `src/db/`       ‚Äî Prisma schema and database helpers
- `src/components/` ‚Äî Reusable React components
- `generated/`    ‚Äî Auto-generated files, do NOT edit manually

## Coding Conventions
- TypeScript everywhere. No `any` types without a comment explaining why.
- Use Prisma for all database access. No raw SQL.
- API routes follow REST conventions. Document with JSDoc.
- Component files: PascalCase. Utility files: camelCase.
- Prefer named exports over default exports.
- Error handling: use our custom `AppError` class in `src/lib/errors.ts`.

## Environment Setup
- Copy `.env.example` to `.env.local` for local development.
- The DATABASE_URL should point to the local Postgres instance.
- Ask a teammate for STRIPE_SECRET_KEY for payments work.

## Do NOT
- Edit anything in `generated/` ‚Äî it is auto-generated by Prisma and will be overwritten.
- Commit directly to `main` ‚Äî always use a feature branch and PR.
- Use `console.log` in production code ‚Äî use the logger at `src/lib/logger.ts`.
- Remove the `v1/` API routes ‚Äî they are deprecated but still serve active clients.

## Testing
- Tests are in `__tests__/` next to source files, or in `tests/` for integration tests.
- We use Vitest. Run `npm test` to run all tests.
- Snapshot tests exist in `tests/snapshots/` ‚Äî update with `npm run test:update-snapshots`.

üí¨ The #remember Command

During a session, you can ask Claude to remember something for next time ‚Äî and it will write it to your CLAUDE.md automatically.

How to use it

# In the Claude Code chat, just say:
Please remember that we use Zod for all API input validation.

# Or more explicitly:
Add to CLAUDE.md: we always use the logger at src/lib/logger.ts, never console.log

Claude will add the note to the appropriate CLAUDE.md file so it's available in future sessions. It's the closest thing to genuine AI learning that doesn't require a PhD to understand.

‚ÑπÔ∏è Claude writes to the most specific applicable CLAUDE.md ‚Äî if you're in a project that has one, it writes there. If not, it falls back to your global one. Commit the file afterward if you want it to stick for your team.

üü¢üîµ Codex & Gemini CLI Equivalents

Both tools have the same concept under different names:

ToolFile nameLocation
Claude Code CLAUDE.md Project root, or ~/.claude/%USERPROFILE%\.claude\
Codex CLI AGENTS.md Project root, or ~/.codex/%USERPROFILE%\.codex\
Gemini CLI GEMINI.md Project root, or ~/.gemini/%USERPROFILE%\.gemini\

Same idea, same file format, different file names. If you use all three tools, you can maintain three files ‚Äî or keep a single source of truth and symlink it: 

# Write once in CLAUDE.md, symlink for the others
ln -s CLAUDE.md AGENTS.md
ln -s CLAUDE.md GEMINI.md
# PowerShell ‚Äî create symbolic links (run as Administrator if needed)
New-Item -ItemType SymbolicLink -Name "AGENTS.md" -Target "CLAUDE.md"
New-Item -ItemType SymbolicLink -Name "GEMINI.md" -Target "CLAUDE.md"
üé≠ Three AI tools, three context file names. The people who named them are very good at programming and somewhat less good at coordination. But symlinks fix everything.

üí° Tips for Writing Good Context Files

1. Write it for a competent stranger

Don't explain what a database is. Do explain why you're using Prisma instead of Drizzle, what AppError is, and why there are two API versions. Claude is smart ‚Äî it just doesn't know your stuff.

2. Commands beat descriptions

"Use the logger module" is worse than "Use import { log } from '@/lib/logger' ‚Äî never use console.log". Specifics stick.

3. The Do Not list is the most important section

Pain is a great teacher. Every time Claude does something that makes you flinch, add it to the Do Not list. This file should grow over time.

4. Keep it concise

A 500-line CLAUDE.md is not better than a 50-line one. Context that's buried in noise doesn't get applied. If every line feels important, that's a sign some lines aren't.

5. Commit it

git add CLAUDE.md AGENTS.md GEMINI.md
git commit -m "docs: add AI context files for Claude, Codex, and Gemini"

Your teammates will thank you. New team members who use AI tools will really thank you.

6. Global file: your preferences, not your project's

Your global CLAUDE.md is for personal preferences that apply everywhere: "I prefer functional programming patterns", "always explain your reasoning before writing code", "never suggest switching to a different framework than the one already in use." Keep it to things that are about you, not about a specific project.
