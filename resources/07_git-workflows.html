üìù Commit Messages

The average developer's commit history is a fossil record of bad decisions and optimism. "fix", "wip", "asdf", "finally". AI changes this ‚Äî not by being creative, but by actually reading what changed.

Ask Claude to write the message

# Stage your changes, then ask Claude:
Please write a git commit message for my staged changes.
Follow conventional commits format (feat/fix/refactor/docs/etc).
Focus on the WHY, not just the what.

Or use the /commit skill (built into Claude Code)

# Claude Code ships with a built-in /commit skill
/commit

This runs git status, git diff, and the recent log, then writes and commits with an appropriate message.

Conventional commits cheat sheet

PrefixUse forExample
feat:New featuresfeat: add OAuth login with Google
fix:Bug fixesfix: prevent double-submit on payment form
refactor:Code changes, no behaviour changerefactor: extract auth logic into useAuth hook
docs:Documentation onlydocs: add MCP setup instructions to README
test:Tests onlytest: add coverage for payment edge cases

Custom commit skill

Create a skill that enforces your team's commit message style every time:

# .claude/skills/commit/SKILL.md
---
name: commit
description: >
  Use when the user asks to commit staged changes.
  Writes a conventional-commit message and commits.
  Do NOT use for general git questions.
---

Write and execute a git commit for the currently staged changes.

Steps:
1. Run `git diff --cached` to see what's staged
2. Run `git log --oneline -5` to match recent commit style
3. Write a message following Conventional Commits:
   - Format: `type(scope): short description`
   - Types: feat, fix, refactor, docs, test, chore, perf
   - Body: explain the WHY, not just the what
   - Max 72 chars for the subject line
4. Run `git commit -m "..."` with the message
5. Confirm success

üîÄ Pull Request Descriptions

A good PR description is the gift that keeps on giving: faster reviews, easier blame archaeology, and a searchable record of decisions. They're also the thing nobody writes because "the code speaks for itself." (It doesn't. It whispers cryptically and lies.)

Ask Claude to write the PR body

# Give Claude the context it needs
Write a PR description for the changes in this branch vs main.
Include: what changed, why it changed, how to test it, and any screenshots if relevant.
Format it in Markdown with ## Summary, ## Changes, and ## Testing sections.

üîç Code Review with AI

AI code review isn't a replacement for human review ‚Äî it's the filter that catches the obvious things so your human reviewers can focus on the interesting things. Like whether this architecture makes sense. Or why there's a sleep(1000) on line 47.

Review before pushing

# Review your own diff before anyone else sees it
Review the staged changes in this PR for:
- Correctness and obvious bugs
- Security issues (injection, exposed secrets, missing validation)
- Performance concerns
- Missing error handling
- Anything that would fail code review

Review a PR from GitHub

# Fetch and review a specific PR
gh pr diff 123 | claude -p "Review this diff for bugs, security issues, and style problems. Be specific and cite line numbers."

---
name: deep-review
description: >
  Use when user asks for a thorough code review of staged changes or a specific file.
  More comprehensive than /review-pr.
  Do NOT use for general questions about code quality.
---

Do a thorough code review of $ARGUMENTS (or staged changes if no argument given).

## Review checklist

**Correctness**
- Does the code do what it claims?
- Are there off-by-one errors, null pointer risks, race conditions?

**Security**
- SQL injection, XSS, CSRF, path traversal
- Hardcoded secrets or API keys
- Missing input validation
- Insecure direct object references

**Performance**
- N+1 queries
- Missing pagination on list endpoints
- Unnecessary re-renders or recomputation

**Maintainability**
- Is this readable in 6 months?
- Are edge cases handled and documented?

## Output format
- Overall risk score: üü¢ Low / üü° Medium / üî¥ High
- List each issue: file:line | severity | description | suggested fix
- End with: "Ready to merge" or "Needs changes before merging"

ü™Ñ Git Skills Worth Creating

Here are some ready-to-use skill templates for common git tasks:

/standup ‚Äî write today's standup from your commits

# ~/.claude/skills/standup/SKILL.md
---
name: standup
description: >
  Use when user asks for a standup summary, daily update, or status report.
---

Generate a standup update from recent git activity.

Steps:
1. Run: `git log --since="yesterday" --oneline --author="$(git config user.name)"`
2. If nothing since yesterday, try: `git log --since="3 days ago" --oneline`
3. Summarize the commits into plain English:
   - What was completed
   - What is in progress
   - Any blockers (infer from WIP commits or incomplete features)

Format as bullet points. No jargon. Write it like you're telling a teammate, not writing a report.

/changelog ‚Äî summarize changes for a release

# .claude/skills/changelog/SKILL.md
---
name: changelog
description: >
  Use when user asks for a changelog, release notes, or version summary.
  Takes a git tag or commit range as argument.
---

Generate a changelog for $ARGUMENTS (or since last tag if no argument).

1. Run: `git log --oneline [tag]..HEAD` (or `git log --oneline $(git describe --tags --abbrev=0)..HEAD`)
2. Group commits by type (feat, fix, refactor, etc.)
3. Write user-facing descriptions ‚Äî not "refactor auth module" but "Login is now faster"
4. Format as Keep a Changelog format:
   ### Added
   ### Changed
   ### Fixed
   ### Removed

/branch ‚Äî create a well-named branch

# ~/.claude/skills/branch/SKILL.md
---
name: branch
description: >
  Use when user wants to create a new git branch with a good name.
  Argument is a description of what the branch is for.
---

Create a new git branch based on: $ARGUMENTS

Branch naming rules:
- Format: type/short-description (e.g. feat/oauth-login, fix/double-submit-bug)
- Lowercase, hyphens not underscores, max 40 chars
- Types: feat, fix, refactor, docs, chore

Steps:
1. Propose 3 branch name options
2. Ask the user to confirm or choose
3. Run: `git checkout -b [chosen-name]`
4. Confirm creation

ü™ù Hooks for Git Automation

Combine Claude Code hooks with git for a fully automated quality gate:

Run tests before every commit (pre-commit equivalent)

# In .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          // Block git commit if tests fail
          "command": "bash -c 'echo \"$CLAUDE_TOOL_INPUT\" | grep -q \"git commit\" && npm test --bail || true'"
        }]
      }
    ]
  }
}

Auto-lint after file edits, before commit

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{ "type": "command", "command": "npm run lint --silent 2>&1 | head -20" }]
      }
    ]
  }
}
üé≠ Traditional pre-commit hook: runs once at commit time, you remember to commit after five hours of flow state.
Claude Code hook: runs every time Claude edits a file, catches problems immediately while Claude is still mid-task and can fix them. This is better.
