üß† What is MCP?

Model Context Protocol is an open standard created by Anthropic that lets AI models talk to external tools, services, and data sources in a structured, secure way. Think of it as a universal adapter between your AI and the rest of the world.

Without MCP, your AI assistant knows only what's in the conversation. With MCP, it can query your database, read GitHub issues, search the web, inspect running processes ‚Äî anything you connect it to.

The three pieces

RoleWhat it doesExample
Host The AI app you're using ‚Äî it manages connections Claude Code, Claude Desktop
Client Built into the host ‚Äî handles the protocol layer Claude's internal MCP client
Server The thing that exposes tools to the AI A filesystem server, a GitHub server, your custom server
‚ÑπÔ∏è Key insight: MCP servers expose tools (actions like "read file", "create issue") and resources (data like "list of open PRs"). Claude calls these the same way it calls built-in tools. From Claude's perspective, it's all just "things I can do."

ü§î Why should I care?

Because copy-pasting context into a chat window is a job, and you already have one of those.

  • üóÑÔ∏è Live data. Query your actual database instead of describing the schema from memory (incorrectly).
  • üêô GitHub integration. Claude can read issues, PRs, and comments ‚Äî and create them ‚Äî without you leaving the terminal.
  • üåê Web access. Let Claude fetch URLs and search the web without manual pasting.
  • üìÇ Extended filesystem access. Give Claude access to folders outside your project directory.
  • üß† Persistent memory. Some MCP servers give Claude a long-term memory that survives across sessions.
  • üîß Custom tools. Build a server that wraps your internal APIs. Now Claude knows your whole stack.
üé≠ Without MCP: "Claude, our users table has columns id, email, created_at, is_active, stripe_customer_id, and..."
With MCP: "Claude, look at the database."

‚ûï Adding MCP Servers to Claude Code

Claude Code has a built-in CLI for managing MCP servers. There are two transport types: stdio (a local process Claude spawns) and SSE (a remote HTTP server).

Add a stdio server (most common)

# The general pattern
claude mcp add <name> -- <command> [args...]

# Example: filesystem server with access to your home directory
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem ~/

# Example: GitHub server
claude mcp add github -- npx -y @modelcontextprotocol/server-github

Add a remote SSE server

# Point Claude at a running HTTP server
claude mcp add --transport sse <name> <url>

# Example: a locally running custom server
claude mcp add --transport sse my-api http://localhost:3000/mcp

Manage your servers

# List all configured servers
claude mcp list

# Show details for one server
claude mcp get <name>

# Remove a server
claude mcp remove <name>

Scoping: project vs. personal

By default, servers are added to your personal config and available everywhere. Use --scope project to save to the current project only (great for team-shared servers): 

# Save to project config (committed to git, shared with team)
claude mcp add --scope project filesystem -- npx -y @modelcontextprotocol/server-filesystem .

# Save to personal config (default ‚Äî available in all projects)
claude mcp add --scope user filesystem -- npx -y @modelcontextprotocol/server-filesystem ~/
‚úÖ Rule of thumb: Servers that make sense for everyone on the project (like a project-specific DB or API) go in --scope project. Servers for personal preference (like your global web search) go in --scope user.

Where the config lives

Scopeüçé macOS pathü™ü Windows path

What the JSON looks like

  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/you/projects"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_your_token_here"
      }
    },
    "my-api": {
      "transport": "sse",
      "url": "http://localhost:3000/mcp"
    }
  }
}
‚ö†Ô∏è Don't commit API keys. If your project-scoped settings file contains secrets, add .claude/settings.json to .gitignore. Or use environment variables and reference them in your shell profile instead.

üî® Building Your Own MCP Server

If an existing server doesn't scratch your itch, you can build your own in about 30 lines of code. The SDK handles all the protocol-level plumbing.

Quick-start with the TypeScript SDK

# Install the SDK
npm install @modelcontextprotocol/sdk

Minimal server example

// server.ts ‚Äî a minimal MCP server with one tool
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

const server = new Server(
  { name: "my-server", version: "1.0.0" },
  { capabilities: { tools: {} } }
);

// Register a tool
server.setRequestHandler("tools/list", async () => ({
  tools: [{
    name: "greet",
    description: "Say hello to a person",
    inputSchema: {
      type: "object",
      properties: { name: { type: "string" } },
      required: ["name"]
    }
  }]
}));

server.setRequestHandler("tools/call", async (req) => ({
  content: [{ type: "text", text: `Hello, ${req.params.arguments.name}!` }]
}));

await server.connect(new StdioServerTransport());

Connect it to Claude Code

claude mcp add my-server -- npx ts-node server.ts
